# План внедрения reliability hardening

Документ фиксирует практический план внедрения идей из дизайн-дока
`RELIABILITY_HARDENING_DESIGN` с учетом текущего состояния CHINT ETM MDM.

Цель: повысить надежность изменяющих операций без поломки уже работающих
сценариев:

- заполнение `_filled.xlsx` с красными обязательными ячейками;
- импорт ручных дозаполнений из `_filled.xlsx`;
- импорт прайса;
- импорт ETIM;
- разбор ETIM-конфликтов через Excel-отчет;
- откат базы из бэкапа;
- portable-сборка через `build_windows.bat`.

## Критика исходного дизайн-дока

Дизайн-док правильно подсвечивает главные риски: прямое копирование SQLite,
конкурентные фоновые операции, доверие к Excel-отчетам, неатомарная запись JSON
и слабая валидация входных файлов.

Но в текущем виде документ слишком широкий для одного цикла разработки.
Если внедрять все сразу, получится большая архитектурная переделка с высоким
риском регрессий. Часть пунктов стоит упростить или отложить.

### Что принять почти без изменений

- Запрет параллельных write-операций в GUI.
- Единая фабрика SQLite-подключений с `foreign_keys` и `busy_timeout`.
- Бэкап через SQLite Backup API вместо `shutil.copy2`.
- Безопасный restore с safety backup и проверкой `integrity_check`.
- Атомарная запись `manual_values.json`, `attribute_mappings.json` и config.
- Сохранение VBA для `.xlsm`.
- Политика заполнения `EMPTY_ONLY` по умолчанию.
- Защита от повторного применения старого ETIM conflict report.
- Проверка `$LASTEXITCODE` в `build_windows.ps1`.

### Что упростить на первом проходе

- Не вводить сразу полноценные `models.py`, `validation.py`, `database.py` и
  большой transaction service. Сначала можно добавить маленькие сервисные
  модули только под реальные задачи: `db_utils.py`, `atomic_files.py`,
  `jobs.py`.
- Не делать большой staging-слой для всех импортов сразу. Для прайса достаточно
  `validate -> backup -> transaction -> report`. Для ETIM staging нужен позже,
  когда будем защищать conflict workflow.
- Не вводить HMAC для строк ETIM-отчета на первом шаге. Достаточно `_meta`,
  `import_id`, исходных значений и compare-and-swap. HMAC можно добавить после
  появления таблицы конфликтов.
- Не добавлять `OVERWRITE_ALL` в GUI сразу. Для текущего процесса достаточно
  `EMPTY_ONLY`; остальные политики оставить как внутренний enum/CLI-флаг позже.
- Не делать полноценный read-only diagnostic mode на первом этапе. Достаточно
  понятной ошибки миграции и отказа запускать изменяющие операции.

### Что отложить

- Полный `import_runs` + аудит всех измененных значений.
- Сложная идемпотентность всех импортов по SHA-256.
- Глубокая переработка ETIM с объединением нескольких строк артикула.
- GUI для удаления manual override.
- `mypy`/`pyright`, пока кодовая база еще активно меняется.
- Мульти-ОС CI. На ближайшем этапе важнее Windows + pytest + сборка.

## Инварианты для совместимости

Эти правила нельзя нарушать при hardening-работах:

1. `_filled.xlsx` остается главным файлом для ручного дозаполнения.
2. Красная подсветка обязательных пустых ячеек сохраняется.
3. Импорт ручных дозаполнений продолжает читать старый формат
   `manual_values.json`.
4. Старые `attribute_mappings.json` продолжают читаться.
5. Старые базы без новых служебных таблиц открываются через миграцию.
6. Старые ETIM-отчеты либо применяются старым безопасным путем, либо
   отклоняются с понятным сообщением "пересоздайте отчет"; молча применять их
   как новые нельзя.
7. Существующие CLI-флаги не удаляются без совместимого alias.
8. Portable-сборка остается пользовательским способом распространения.

## Этап 1. Быстрые защитные правки без изменения схемы

Цель: закрыть самые опасные риски без миграций базы.

### REL-001. Блокировка write-операций в GUI

Сделать простой `JobController` или минимальный слой в `app.py`, который:

- не позволяет запустить второй импорт/restore во время активной write-операции;
- блокирует кнопки импорта прайса, импорта ETIM, применения ETIM-решений и
  restore;
- держит сильную ссылку на активный worker;
- возвращает кнопки в нормальное состояние и при успехе, и при ошибке.

Совместимость: пользовательский интерфейс остается прежним, меняется только
поведение кнопок во время операции.

Проверки:

- двойной клик по `Импортировать ETIM` запускает один worker;
- restore недоступен во время импорта;
- после ошибки кнопки снова активны.

### REL-002. SQLite Backup API

Заменить текущие `shutil.copy2`-бэкапы на согласованный backup:

- открыть источник read-only;
- выполнить `source.backup(destination)`;
- проверить `PRAGMA integrity_check`;
- сохранить уникальное имя с микросекундами;
- вернуть путь бэкапа.

Сначала использовать новый backup в:

- импорте прайса;
- импорте ETIM;
- применении ETIM-решений;
- restore safety backup.

Совместимость: расширение и смысл `.bak_*` сохраняются. Старые бэкапы
остаются пригодными для restore после проверки целостности.

### REL-003. Безопасный restore

Restore должен:

- проверять выбранный бэкап через `integrity_check`;
- создавать safety backup через SQLite Backup API;
- не заменять основную базу, если бэкап битый;
- по возможности закрывать активные соединения до замены файла;
- не трогать `-wal`/`-shm`, пока не понятно, что соединения закрыты.

На первом этапе можно оставить файловую замену через `os.replace`, но только
после проверки временной восстановленной базы.

### FIL-001. `EMPTY_ONLY` для заполнения

Текущее заполнение пишет значения в желтые колонки без отдельной политики.
Нужно зафиксировать правило:

- если ячейка уже непустая, программа ее не перезаписывает;
- ручные значения из `manual_values.json` применяются только к пустой ячейке;
- расхождение непустой ячейки с базой попадает в технический отчет.

Совместимость: для обычных пустых шаблонов поведение не меняется. Для уже
частично заполненных шаблонов поведение станет безопаснее.

### XLS-001. Сохранение VBA для `.xlsm`

Для `.xlsm` использовать `load_workbook(..., keep_vba=True)`.
После сохранения проверить, что `xl/vbaProject.bin` не потерян, если был во
входном файле.

Совместимость: `.xlsx` работает как раньше.

### AFS-001. Атомарная запись JSON/config

Добавить `atomic_files.py`:

- `atomic_write_text`;
- `atomic_write_json`;
- `unique_output_path`.

Перевести на атомарную запись:

- `manual_values.json`;
- `attribute_mappings.json`;
- `config.json`.

Совместимость: формат JSON не меняется.

### BLD-001. Hardening build script

В `build_windows.ps1` добавить:

- проверку `$LASTEXITCODE` после `pip` и PyInstaller;
- удаление старого zip перед упаковкой уже есть, оставить;
- smoke check наличия exe уже есть, оставить;
- опционально запуск `--version` или импорт пакета до PyInstaller.

Совместимость: команда сборки остается `build_windows.bat`.

## Этап 2. Валидация входных файлов и ручных значений

Цель: не начинать изменяющую операцию, если входной файл явно не тот.

### PRI-001. Валидация прайса перед бэкапом

Перед импортом:

- проверить расширение `.xlsx/.xlsm`;
- проверить ZIP-структуру;
- проверить наличие листа `Price-list`;
- нормализовать заголовки;
- проверить обязательные заголовки `Артикул`, `Наименование`;
- проверить дубли заголовков после нормализации.

Бэкап создавать только после успешной валидации.

Совместимость: корректные текущие прайсы импортируются как раньше.

### SEC-001. Безопасное скачивание прайса

Минимальный набор:

- разрешить только `https`;
- скачивать в `.part`;
- ограничить размер;
- очищать имя файла через `Path(...).name`;
- проверять, что итоговый путь внутри `downloads/price`.

Совместимость: кнопки в GUI не меняются.

### MAN-001. Ручные значения без смены формата

Сначала не вводить `active/deleted`. Вместо этого:

- оставить текущий формат `manual_values.json`;
- добавить атомарную запись;
- добавить защиту от дублей `(article, field)`;
- в отчете импорта явно показывать `imported` и `skipped`.

Удаление override сделать отдельным этапом после стабилизации.

## Этап 3. ETIM conflict safety

Цель: старый или отредактированный отчет не должен затереть новые данные.

### ETI-001. Метаданные ETIM-отчета

Добавить скрытый лист `_meta`:

- `report_version`;
- `created_at`;
- `db_path_hint`;
- `import_id`, если уже есть;
- `source_etim_file`;
- `source_etim_sha256`.

На первом проходе не нужен `database_id`, если в базе его еще нет.

Совместимость: старые отчеты без `_meta` отклонять с понятным сообщением:
"Отчет создан старой версией, пересоздайте ETIM-импорт".

### ETI-002. Compare-and-swap для конфликтов

В листе `Конфликты` сохранить исходное значение базы на момент отчета.
При применении:

- проверить, что текущее значение в базе все еще равно исходному;
- если отличается, строку пропустить как устаревшую;
- применить только строки `Принять ETIM`;
- все изменения выполнить одной транзакцией.

Совместимость: пользователь по-прежнему выбирает `Оставить базу` или
`Принять ETIM`.

### ETI-003. Габариты без угадывания порядка

Ужесточить запись общего размера:

- независимые колонки длина/ширина/высота можно писать как сейчас;
- общий размер писать только при понятном порядке;
- неоднозначное значение не писать автоматически, а показывать в отчете.

Совместимость: уже записанные значения не трогать.

## Этап 4. Схема, аудит и идемпотентность

Этот этап стоит делать только после стабилизации первых трех.

### DB-001. `schema_meta`

Добавить простую таблицу:

```sql
CREATE TABLE IF NOT EXISTS schema_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

Хранить:

- `schema_version`;
- `database_id`.

Перед миграцией создавать backup через новый Backup API.

### DB-002. `import_runs`

Добавить журнал импортов:

- тип импорта;
- SHA-256 файла;
- статус;
- backup path;
- report path;
- ошибка, если была.

На первом проходе использовать для прайса и ETIM.

### ETI-004. Таблица ETIM-конфликтов

После появления `import_runs` перенести конфликтные строки в таблицу
`etim_conflicts`. Excel-отчет станет только UI-формой для выбора решения, а
истина будет храниться в базе.

## Этап 5. Тесты и CI

Минимальный набор до релизной точки:

- unit-тест атомарной записи;
- backup/restore на временной SQLite;
- заполнение `.xlsx` с непустой ячейкой в `EMPTY_ONLY`;
- сохранение `.xlsm` VBA, если есть фикстура;
- импорт ручных значений из `_filled.xlsx`;
- ETIM conflict apply со stale-значением;
- Windows CI: pytest + PyInstaller build.

`ruff` можно добавить сразу. `mypy/pyright` отложить до появления новых
сервисных модулей.

## Рекомендуемый порядок ближайших задач

1. `REL-001`: блокировка конкурентных write-операций.
2. `REL-002`: SQLite Backup API.
3. `REL-003`: безопасный restore.
4. `AFS-001`: атомарная запись JSON/config.
5. `FIL-001`: `EMPTY_ONLY`.
6. `XLS-001`: `.xlsm keep_vba`.
7. `PRI-001` + `SEC-001`: безопасный прайс.
8. `ETI-001` + `ETI-002`: защищенные ETIM-отчеты.
9. `DB-001` + `DB-002`: миграции и журнал импортов.
10. `TST-001`: тесты и CI.

## Что не делать сейчас

- Не переписывать все импорты на новую архитектуру одним коммитом.
- Не вводить большой staging для прайса до базовой валидации.
- Не ломать старый `manual_values.json`.
- Не менять рабочий процесс ПМ: `_filled.xlsx` остается основным файлом.
- Не возвращать отдельный PM-report workflow.
- Не делать упаковочные поля ближайшим приоритетом.

## Definition of Done для первого этапа

Первый этап можно считать завершенным, когда:

- write-операции не запускаются параллельно из GUI;
- все изменяющие операции создают backup через SQLite Backup API;
- restore не заменяет базу при битом бэкапе;
- JSON/config пишутся атомарно;
- заполнение не перезаписывает непустые пользовательские ячейки;
- `.xlsm` не теряет VBA;
- portable build script корректно падает при ошибках pip/PyInstaller;
- README обновлен только там, где меняется пользовательское поведение;
- есть хотя бы минимальные тесты на backup/restore, `EMPTY_ONLY` и atomic JSON.
